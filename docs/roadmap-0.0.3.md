# 0.0.3 Roadmap

本文把 `0.0.3` 的目标拆成更清晰的实现顺序，方便后续继续迭代时保持节奏稳定。

## 总目标

`0.0.3` 不追求大幅扩张命令面。

这一版更适合聚焦三件事：

- 让 `find` 的共享语义更集中、更不容易漂移
- 让 `hybrid` 的 fallback 判断更稳定、更可测试
- 让写命令失败时的诊断信号更适合脚本和 agent 消费

## 为什么这样排序

推荐把这一版排成“读语义 -> fallback 信号 -> 写错误”的顺序，原因是：

1. `find` 语义是读路径里最容易继续分叉的部分，越早收敛越省后续维护成本。
2. fallback 能力信号会影响 `hybrid` 路径上更多读命令的可预测性，适合作为第二阶段稳定项。
3. 写命令诊断属于非常重要的体验项，但对整体架构牵引较小，放在后面更容易收尾。

## 阶段拆分

### 阶段 1：收敛 `find` 共享语义

目标：

- 统一日期过滤语义
- 统一标签过滤语义
- 统一默认可见条目策略

建议工作：

- 盘点 CLI 层、backend 层、local SQL 后处理里与 `find` 语义相关的逻辑
- 明确哪些逻辑属于“共享语义”，哪些属于“backend 特定查询策略”
- 把共享语义沉到复用层，避免 `web` / `local` 分别维护一套近似实现

建议先补的测试：

- 同一组 `--date-after` / `--date-before` 输入在不同读路径下的结果边界
- 多个 `--tag` 与 `--tag-any` 的组合行为
- attachment / note / annotation 的默认可见性

阶段完成标志：

- `find` 的共享语义定义集中在更少的位置
- 新增过滤条件时不需要同时改多处近似逻辑

### 阶段 2：稳定 fallback 能力信号

目标：

- 让 `hybrid` 的 fallback 判断建立在稳定的能力信号上
- 减少对错误文案或消息片段的依赖

建议工作：

- 盘点当前仍依赖错误文本判断的 fallback 路径
- 抽出更清晰的“可 fallback / 不可 fallback / 能力缺失 / 数据缺失”信号
- 让测试直接断言行为和错误类型，而不是盯住文案

建议先补的测试：

- local backend 能力不支持时，`hybrid` 是否按预期回退
- 输入错误或显式模式边界错误时，`hybrid` 是否拒绝错误回退
- fallback 后返回结果是否仍符合现有 JSON 合同

阶段完成标志：

- `hybrid` 关键路径不再依赖脆弱的错误字符串
- fallback 行为能用更稳定的测试覆盖住

### 阶段 3：补强写命令可诊断性

目标：

- 让常见写命令失败原因更容易被脚本和 agent 区分

建议工作：

- 盘点 create / update / delete 系列命令现在还偏模糊的报错
- 区分参数错误、输入文件错误、JSON 结构错误、缺失前置条件、权限开关阻止等失败类别
- 让错误输出尽量保持稳定，避免未来测试和自动化依赖易变文案

建议先补的测试：

- 缺少必需前置版本
- `--data` / `--from-file` 冲突
- 文件不存在或不可读
- JSON 合法但结构不符合要求
- 权限开关阻止写入或删除

阶段完成标志：

- 常见失败场景能被稳定地区分
- 脚本不需要靠模糊文本猜测失败原因

## 建议的执行顺序

如果按较稳的节奏推进，推荐顺序是：

1. 先为 `find` 共享语义补测试
2. 再收敛 `find` 的共享实现
3. 然后补 `hybrid` fallback 行为测试
4. 再清理 fallback 信号模型
5. 最后补写命令错误测试并完善报错输出

## 这一版不急着做的事

`0.0.3` 不建议把范围扩得太大，下面这些更适合留到后续版本：

- local full-text search
- MCP server 支持
- 大规模命令面扩张
- 本地数据库写入

## 预期结果

如果 `0.0.3` 按这个顺序完成，项目会得到三个直接收益：

- 新增读能力时更不容易把 `web` / `local` / `hybrid` 弄出细小分叉
- `hybrid` 的行为会更容易预测，也更容易 review
- 写命令失败时更适合自动化系统做恢复、重试和提示
